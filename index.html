New Rule T1053.005 Scheduled Task Creation via schtasks.exe
⚙ Context appliedSCCM ConfigMgr_* task name exclusion pre-loaded from TH-2025-091 · scoped to index=sysmon on Corp domain · msiexec/setup parent excluded as env baseline
SPL · T1053.005
| Gap fill: T1053.005 Scheduled Task · DetectionLogicAgent
index=sysmon EventCode=1 Image="*\\schtasks.exe"
    CommandLine="*/create*"
| rex field=CommandLine "\/tn\s+(?<task_name>\S+)"
| rex field=CommandLine "\/tr\s+(?<task_cmd>.+?)(\s+\/|$)"
| where NOT match(ParentImage,"(msiexec|setup|install)")
| stats count by _time, host, User, task_name, task_cmd, ParentImage
| eval risk=if(match(task_cmd,"(powershell|cmd|wscript|cscript|rundll32)"),"HIGH","MED")
| table _time, host, User, task_name, task_cmd, ParentImage, risk

❓ Frequently Asked Questions
Platform guidance, workflow tips, and agent & tool reference
Getting Started
What is PrimeTH and what does LOCK stand for?

PrimeTH is a structured threat hunting platform built around the LOCK framework — a four-stage methodology for proactive adversary detection:

  • L — Learn: Ingest and analyse CTI, extract TTPs, and form hypotheses.
  • O — Observe: Define the environment baseline — what normal looks like, what suspicious looks like, and which observables to watch for.
  • C — Check: Write and test SPL detection logic against Splunk Enterprise Security.
  • K — Keep: Capture findings, build a timeline, log evidence, and produce the hunt report.

Each stage is supported by specialised AI agents that reason over tradecraft, data sources, and detection logic in parallel.

How do I create a new hunt?

Click + New Hunt in the top-right corner of the screen. Fill in the hunt title, select a primary TTP focus, and choose a lead analyst. The hunt is created in Draft status and opens directly into the Learn stage where you can paste CTI and start TTP extraction.

Hunts progress through Draft → Active → Closed as you gate each LOCK stage.

How do I switch between hunts without losing my place?

Use the hunt switcher dropdown in the hunt detail header (the monospace hunt ID button, e.g. TH-2026-041). Switching hunts preserves your current LOCK stage — if you were on Check in hunt 041 and switch to hunt 040, you stay on the Check tab.

The exception is the Keep stage — if the destination hunt has no Keep data yet, the platform falls back to the Learn tab.

The LOCK Workflow
What happens in each LOCK stage?
  • Learn: Paste a CTI report or threat brief. The Hypothesis Agent extracts TTPs, maps them to MITRE ATT&CK, and proposes hunt hypotheses. The Tradecraft and Detection Logic agents then load relevant context and draft SPL stubs.
  • Observe: Define and document your environment baseline for this hunt. Edit what normal looks like, what suspicious looks like, and the expected observables per category (Processes, Network, Files, Authentication). Sub-hunts each have their own Observe profile.
  • Check: Run SPL queries in the embedded query runner, review RAA (Reasoning Augmented Analytics) findings, and track hypothesis coverage. The Check Summary card rolls up RAA + query results to show which gaps have been addressed.
  • Keep: Log confirmed findings, attach evidence, build a timeline of adversary activity, and generate the final hunt report for stakeholders.
What are sub-hunts and when should I use them?

Sub-hunts let you decompose a single hunt into per-TTP investigation threads. Each sub-hunt has its own Observe profile (normal/suspicious/observables) and its own SPL queries in the Check stage.

Use sub-hunts when a hunt covers multiple distinct TTPs that require different data sources or detection approaches — for example, splitting T1570 (Lateral Tool Transfer) and T1003.001 (LSASS Credential Dumping) into separate sub-hunts lets each analyst focus on one thread without cluttering the shared view.

Select a sub-hunt from the sidebar that appears on the left of the Hunt Detail pane when sub-hunts are configured.

What is the Observe stage used for — can I skip it?

The Observe stage is your hunt-specific baseline document. It tells the agents (and the team) what "normal" looks like in this environment for the scope of this hunt, and what adversary artifacts to watch for.

You can technically skip it, but agents in the Check stage use Observe data to calibrate thresholds, choose exclusions, and interpret RAA findings. Skipping it means agents fall back to generic defaults, which increases false positives.

Click ✏ Edit in the Observe tab to update any item — existing entries become inline text fields, and you can add or remove entries per section.

What are gate decisions and where do I find them?

A gate decision is a formal record of whether a LOCK stage has been passed or escalated — who made the call, when, and why. They are logged in kb/gate-decisions.js keyed by hunt ID.

Gate decisions appear in the Hunt Detail header as status chips, and in the hunt history overlay (clock icon). They provide an audit trail for each stage transition, useful for post-hunt reviews and compliance reporting.

Agents & Tools
What are the five agents and what does each one do?
  • 🎛️ Orchestrator Agent — spawns and coordinates all other agents; manages task routing and pipeline state.
  • 💡 Hypothesis Agent — runs first; analyses CTI, extracts TTPs, and formulates testable hypotheses. Completes before the others start.
  • 🗄️ Data Engineering Agent — identifies required log sources, validates index availability, and ensures the data pipeline is ready for query.
  • 🧠 RAA Supervisor Agent (Tradecraft) — applies threat tradecraft knowledge to interpret anomalies and flag process-chain or command-line deviations that rule-based queries may miss.
  • ⚙️ Detection Logic Agent — drafts, tunes, and validates SPL detection rules; integrates FP exclusions from the Observe baseline.

Click any agent card in the Agents tab or the Observe panel to open its detail drawer.

What is RAA (Reasoning Augmented Analytics)?

RAA is an always-on analytic layer in the Check stage that runs alongside SPL queries rather than replacing them. It applies two lenses:

  • Process Chain Anomaly — detects unusual parent-child process relationships that deviate from known-good baselines.
  • Command Line Anomaly — flags suspicious argument patterns (LOLBin abuse, encoded commands, unusual flags) in process execution telemetry.

RAA findings appear at the top of the Check stage. The Check Summary card below RAA shows whether your SPL queries have addressed the specific gaps RAA identified — if a RAA finding has no corresponding query, the gap is flagged as open.

Not all RAA findings will be applicable to every hunt; use analyst judgement when interpreting results.

What is the Rule Validation tool and how is it different from running a query?

Rule Validation (✅ in the Tools Available sidebar) is a static analysis tool — it reviews SPL logic for syntax correctness, field name validity, and detection coverage without executing against live data.

Running a query in the Check stage query runner executes against Splunk ES and returns real results. Rule Validation is the pre-flight check; the query runner is the live test.

Rule Validation is a tool, not an agent — it does not appear in the agent topology, feed filter pills, or agent model list.

Detection & Analysis
How do I read the Check Summary card?

The Check Summary card appears at the top of the Check stage and has two states:

  • Pre-run: Shows the RAA findings count and how many hypotheses have been mapped to queries. Prompts you to run queries to complete the assessment.
  • Post-run: After at least one query is executed, the card shows a combined assessment — which RAA gaps were addressed by the query results, and which remain open. An overall verdict (Covered / Partial / Gap) is assigned per hypothesis.

The summary is per-hunt and resets if you switch to a different hunt. It does not persist between sessions.

Why are some TTPs shown in red, yellow, or indigo on the MITRE grid?

The colour coding on the MITRE ATT&CK grid in the Learn stage reflects detection confidence for each TTP in the context of this hunt:

  • Red (chip-red) — confirmed or highly likely adversary activity; primary detection target.
  • Indigo (chip-indigo) — possible or supporting TTP; secondary target.
  • Yellow — low confidence or insufficient data.

Hover over any TTP chip to see the full technique name, tactic, and a brief description.

How do I tune a query to reduce false positives?

False positive tuning follows a three-step approach in PrimeTH:

  • 1 — Observe stage: Document known-good behaviour under "What Normal Looks Like" and list approved processes/accounts/IPs in Expected Observables. Agents use this to auto-suggest exclusions.
  • 2 — Check stage exclusions: Modify the SPL | where NOT match(...) or | search NOT clauses directly in the query editor. The Detection Logic Agent will suggest tuned versions based on the Observe baseline.
  • 3 — Runbooks: Each TTP runbook in the Knowledge Base has an Exclusions section listing pre-validated FP patterns for that technique. Open the runbook from the KB tab to copy relevant exclusions.
Knowledge Base
What is the difference between Platform Attack Techniques and Runbooks?

Platform Attack Techniques are detection packages tied to a specific technology platform or log source not yet fully onboarded into the environment — for example, VMware vCenter, Ivanti Connect Secure, or SolarWinds Orion. Each skill includes detection patterns, an SPL template, FP exclusions, and attack path context for that platform.

Runbooks are technique-level playbooks keyed to MITRE ATT&CK TTP IDs (e.g. T1003.001). They are platform-agnostic and cover the full investigation procedure: data sources, SPL queries, triage steps, and response actions.

Use Platform Attack Techniques when onboarding a new log source; use Runbooks as the step-by-step investigation guide during an active hunt.

What is the IOC Repository?

The IOC Repository (previously IOC Repository) in the Knowledge Base stores indicators and artifacts associated with past and active hunts — IP addresses, file hashes, domain names, JA3 fingerprints, registry keys, and similar forensic artifacts.

Artifacts are defined in kb/iocs.js and are searchable from the KB search bar. They can be referenced in hunt findings and linked to specific TTPs or adversary campaigns.

How do I add a new Platform Technique or Runbook?

Both are defined in Markdown source files that serve as the runtime source of truth:

  • Platform Technique: Open kb/skills.md, copy any existing ## SK-XXX — block, paste it above the final ---, fill in the fields, then mirror the entry in kb/skills-fallback.js (used for local file:// development).
  • Runbook: Open kb/runbooks.md, copy any existing ## T… block, paste it, and fill in the TTP ID, data sources, SPL, and triage steps.

Required fields for a skill: type, category, category-label, ttps, author. No build step is required — changes are picked up on next page load when serving over HTTP.
